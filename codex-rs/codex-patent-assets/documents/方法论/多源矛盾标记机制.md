# 多源矛盾标记机制设计

> 本知识库有四大教义源讨论同一主题，需要标记它们之间的一致和分歧。

---

## 背景

专利实务知识库有四个独立来源讨论相同法律主题：

| 来源 | 简称 | 前缀 | 特点 |
|------|------|------|------|
| 审查指南 | 审查 | `审查-*` | 官方标准，权威性最高 |
| 以案说法 | 案例 | 无前缀 | 复审委员会实务案例 |
| 崔国斌《专利法》 | 原理 | `原理-*` | 学理分析+法院案例 |
| 尹新天《详解》 | 法条 | `法条-*` | 立法原意+条文解析 |

当这四个来源对同一问题有不同观点时，当前仅列出交叉引用，未标注"一致"或"分歧"。

## 矛盾类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 标准分歧 | 不同来源对同一审查标准有不同表述 | 以案说法认为X标准，审查指南认为Y标准 |
| 适用范围分歧 | 对同一规则的适用范围有不同理解 | 崔国斌认为规则适用于A，审查指南限缩到B |
| 时代差异 | 不同时期观点发生变化 | 2020年修法前后的标准变化 |
| 侧重差异 | 观点一致但侧重不同 | 以案说法侧重审查员视角，崔国斌侧重申请人视角 |

## 标记语法

在 wiki 页面的交叉引用中增加标记：

```markdown
## 关联知识

- **审查标准：** [[审查-创造性-技术启示]] ⟷一致
- **学理分析：** [[创造性-原理-显而易见的认定]] ⟷侧重差异（申请人视角）
- **法条释义：** [[侵权-法条-等同原则]] ⟷标准分歧（认定标准更宽）
- **实务案例：** [[创造性-概述与三步法框架]] ⟷一致
```

### 标记含义

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| `⟷一致` | 两个来源对同一问题表述一致 | 默认标记，无需特别说明 |
| `⟷侧重差异(原因)` | 观点实质一致但表述侧重不同 | 需简要说明差异点 |
| `⟷标准分歧(描述)` | 两个来源有实质性不同 | 需说明具体分歧是什么 |
| `⟷时代差异` | 因修法或审查实践变化导致的不同 | 需标注变化时间点 |

## 实施方式

### 方式一：编译时标记（推荐）

在每次编译新的 wiki 页面时，如果页面涉及的主题在其他来源已有页面，编译者应：
1. 读取相关来源的已有页面
2. 对比核心观点
3. 在交叉引用中添加标记

### 方式二：批量审查

使用 Lint 脚本批量检测潜在矛盾点：
```python
# 伪代码
for topic in cross_referenced_topics:
    pages = get_all_pages_for_topic(topic)
    if len(pages) >= 2:
        check_consistency(pages)  # 提取核心观点，比较是否一致
```

## 优先标记的主题

以下主题最容易产生多源分歧，应优先标记：

| 主题 | 涉及来源 | 典型分歧点 |
|------|---------|-----------|
| 创造性-技术启示 | 审查/案例/原理 | "公知常识"认定标准宽严不一 |
| 等同侵权 | 侵权/法条/判决 | 等同特征三要素的适用 |
| 说明书充分公开 | 审查/案例/原理 | "能够实现"的具体标准 |
| 权利要求解释 | 侵权/审查/判决 | 内部证据vs外部证据的优先级 |
| 现有技术抗辩 | 侵权/案例/判决 | 比对方式（单独对比vs组合） |

## 与现有工作流的整合

1. **编译时：** 编译器（Agent）在写交叉引用时自动标记
2. **Lint时：** `lint_wiki.py` 增加一个检查项：多源页面缺少一致性标记时发出警告
3. **Ingest时：** 新来源到达时，检查是否有相关主题的已有页面，如有则对比并标记
